Chapter 2
System Overview
2.1

Problem Summary

CSEDU currently has no unified digital system. Academic resources are scattered across
flash drives, email threads, and shared folders that nobody can find. Research papers live
on personal laptops. Library operations involve manual ledgers. Students have no visible
place to showcase their projects. And there is zero AI-assisted discovery of any kind.
This platform consolidates four core modules into a single web-based system, layered with
a RAG-based AI Agent that actually understands what’s in the documents — not just
keyword-matches them.
What problem are you solving?
CSEDU currently lacks a unified system to:
• Manage academic resources, research outputs, student projects, and library operations
in one place
• Enable intelligent discovery and semantic understanding of academic content
• Provide users with contextual insights about departmental knowledge and resources
• Leverage modern AI technologies to improve accessibility and reduce information
discovery overhead
Resources are scattered across multiple platforms, library workflows require manual
processes, and users struggle to discover relevant academic materials without significant
search effort. Additionally, there is no intelligent system to answer domain-specific
questions or provide contextual understanding of the institution’s collective knowledge.
Why does it matter?
A unified platform with integrated AI capabilities will:
• Streamline storage, discovery, and access to institutional knowledge
• Reduce administrative overhead for librarians and faculty
• Vastly improve visibility and accessibility of research and academic materials

Project Goals

• Create a centralized digital archive
• Unify four core components into one web platform with integrated AI capabilities
• Enable research publication and sharing with AI-assisted discovery
• Showcase student projects with intelligent curation and recommendation
• Provide a searchable digital library with semantic understanding
• Support multi-format media with AI content analysis capabilities
• Digitize library workflows with AI-assisted patron support
• Implement robust access tiers with context-aware AI responses
• Deploy a RAG-based AI Agent for intelligent knowledge discovery and
synthesis
• Enable natural language querying across all platform modules
• Provide automated summarization of research papers and documents
• Run entirely on free infrastructure during the lab phase (Groq free tier + Gemini free
tier for fallback + minilm for embeddings + Docker + pgvector)

2.4

Out of Scope

The following will NOT be built in this iteration. This is a deliberate list, not a wishlist:
• LMS integration (Moodle, Canvas — not our problem)
• External academic database APIs (Scopus, IEEE Xplore, Google Scholar)
• Plagiarism detection (AI summaries are supplementary, not enforcement tools)
• Real-time user-to-user messaging (AI chat is one-directional by design)
• Custom LLM training from scratch (we use pre-trained models like civilized
engineers)
• Financial accounting beyond library fine tracking
• External payment gateway integration in the MVP (stubbed, tracked internally)

3.1.2

Data Flow Summary

1. All traffic enters through Nginx which routes to frontend, Go API, or RAG service
2. Next.js handles all UI rendering and forwards API calls to Go
3. Go API handles authentication, RBAC enforcement, and all CRUD operations
4. Document uploads go to MinIO (binary) + PostgreSQL (metadata), then a job is
queued in Redis
5. Ingestion Worker picks up the job, extracts text, chunks it, generates embeddings
using MiniLM (sentence-transformers, local), and writes vectors to pgvector
6. User query flow:
• (Optional) Query rewritten using Gemini
• Query embedded using MiniLM
• Hybrid retrieval performed:
– Vector search via pgvector
– Keyword search via PostgreSQL Full-Text Search (FTS)
• Results merged and ranked
• Context sent to Groq LLM
• On failure, fallback to Gemini
• Grounded response returned with citations
The two flows below are the most complex in the system. Both involve async workers,
external APIs, and multi-step pipelines where timing and failure handling matter. Sequence
diagrams are provided to make the ordering and async boundaries explicit.

Why hybrid retrieval (vector + FTS)? Pure vector search is great for semantic
similarity — “papers about neural networks” finds relevant results even if those words
don’t appear verbatim. But it fails on exact lookups: searching for a specific author
name, ISBN, or paper title. Pure FTS is the opposite — exact matches work great,
semantic miss is total. Hybrid retrieval with Reciprocal Rank Fusion gives both:
semantic understanding and exact-match precision. For a platform handling both
casual questions and precise catalog lookups, this is the right call.

The biggest call we made: Using pgvector instead of a dedicated vector database.
At 100K documents with 768-dimensional vectors, pgvector with an HNSW index
gives sub-second ANN (Approximate Nearest Neighbour) retrieval. We benchmarked
this decision: for our scale it holds. If the platform ever hits 1M+ documents, migrate
to Weaviate. The pgvector query interface is identical — it’s a one-config change, not
a rewrite.

4.3

Constraints

Constraints are the rules your database enforces even when your application code has a
bug, gets bypassed, or someone runs a raw SQL query at 2am because production is on
fire. If it’s not in the database, it doesn’t exist. This section documents every meaningful
constraint in the system and why it’s there — not just what it is.

5.1

Flow 1: Document Upload → RAG Ingestion
Pipeline

This is the most technically complex flow and the backbone of the entire AI system. If
this breaks, the AI has nothing to search.
Trigger: A staff member or researcher uploads a file via the frontend.
1. Next.js sends a multipart/form-data POST to Go API: POST /api/v1/documents
2. Go API validates: file type (PDF/video/image/audio), file size (PDF ≤ 100MB,
video ≤ 200MB), and user’s role tier (≥ staff)
3. File binary streamed to MinIO; metadata written to media_items and metadata
tables with status = ’draft’
4. Go API pushes a job to Redis: {"job":
"..."}

"ingest", "item_id":

"...", "file_path":

5. HTTP 201 returned to frontend immediately — the user doesn’t wait for ingestion
6. Ingestion Worker (Go) picks up the Redis job
7. Text extracted from the file using PyMuPDF (called via Python subprocess or RPC)
8. Text split into overlapping chunks (512 tokens, 50-token overlap) to preserve context
across chunk boundaries
9. Each chunk sent to paraphrase-multilingual-MiniLM-L12-v2 (runs locally in
Docker) — returns 768-dim vector
10. Vectors + chunk text written to vector_embeddings table via pgvector
11. media_items.status updated; full-text search index refreshed
12. Success: Document is searchable via keyword search immediately after step 3;
semantic search available after step 10

Failure and edge cases:
• Corrupt PDF: PyMuPDF raises exception → job marked ingestion_failed in
DB → uploader notified via dashboard alert. Document remains accessible as a
binary download; semantic search just won’t work for it.
• Groq API down during ingestion: Doesn’t matter — embeddings are generated
locally. LLM is only called at query time, not ingestion time. This is an intentional
architectural decision.
• File exceeds size limit: Rejected at Go API layer (step 2) before any storage is

6.1

Authentication Approach

• Primary: OAuth 2.0 / SAML SSO via institutional identity provider
• Fallback: Local email + password authentication (always maintained — SSO
providers go down)
• Token mechanism: JWT access tokens (1-hour expiry) + refresh tokens (7-day

• Password hashing: bcrypt with cost factor 12 (never MD5, never SHA1, never
plain text — this is non-negotiable)
• Session invalidation: Refresh tokens stored in user_sessions table; server-side
invalidation on logout
• Brute force protection: Account locked after 5 failed login attempts per minute

6.4

AI-Specific Security

This is the most important security section for this project — the AI access control must
be enforced at the retrieval layer, not post-generation.
The golden rule: Restricted documents must be filtered in the pgvector WHERE
clause before any chunk is returned. They must never enter the LLM prompt. If a
restricted document enters the prompt, the LLM might reference it even if instructed
not to. Defense in depth: filter it out entirely so it never has the chance.
• Access tier filter injected into every pgvector query (see Flow 2, step 6)
• RAG service receives user’s role tier from Go API on every request — it never trusts
the client
• Restricted content queries return HTTP 403 with “Access Denied” before RAG
query is even executed (FR-AI-018)
• AI responses include source citations — every factual claim is traceable and verifiable
(FR-AI-005, NFR-TXX-019)
• AI-generated outputs flagged as supplementary — never authoritative source replacements

don’t call it manually
• Log retention: ≥ 90 days in PostgreSQL (NFR-TXX-017); archived to MinIO beyond
that
• Logs include: actor, action, resource type, resource ID, IP address, timestamp
• audit_log is append-only via PostgreSQL row-level security — even admins cannot
delete audit records

Dockerfile Strategy

Every production Dockerfile uses multi-stage builds. This keeps image sizes small
and attack surfaces minimal.
• Next.js: Stage 1 = Node.js 20 builder (npm run build). Stage 2 = Node.js 20
Alpine runner with only the .next/standalone output. Final image ≈ 200MB.

• Go API: Stage 1 = golang:1.22-alpine (compile to static binary with CGO_ENABLED=0).
Stage 2 = gcr.io/distroless/static-debian12 — minimal attack surface, no
shell, no package manager. Final image ≈ 20MB.
• Workers: Same Dockerfile as Go API, different CMD pointing to the worker binary
entrypoint. No code duplication.
• RAG Service: python:3.11-slim with sentence-transformers + PyMuPDF +
fastapi + uvicorn. The sentence-transformers model is downloaded at build
time and baked into the image so cold start is fast.
